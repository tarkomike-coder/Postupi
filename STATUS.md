# Статус проекта Postupi

Мониторинг поступления Вари (МАИ и Бауманка): автоматический скрейпинг
конкурсных списков, симуляция зачисления (deferred acceptance + Monte
Carlo), дашборд на [postupi.tarko.su](https://postupi.tarko.su).

## Что сделано

### МАИ — полностью автоматизировано
- Скрейпинг публичных конкурсных списков priem.mai.ru без авторизации
  (`backend/services/mai_list_scraper.py`).
- Дополнительный источник для МАИ — публичный JSON API Госуслуг
  (`backend/services/mai_gosuslugi_scraper.py`,
  `backend/services/mai_gosuslugi_import.py`). На Госуслугах МАИ имеет
  `orgId=19`; нужные конкурсные группы фильтруются по ОКСО
  `09.03.01`, `09.03.02`, `09.03.03`, `27.03.03`, `10.05.02`,
  уровню `Базовое высшее образование`, форме `Очная`, типу мест
  `Основные места в рамках КЦП`.
- Источник Госуслуг для МАИ сохраняется отдельно как university
  `МАИ Госуслуги`, а не смешивается с официальным списком priem.mai.ru.
  Это важно: Госуслуги отдают более широкий поток заявлений/статусов,
  а сайт МАИ — опубликованный конкурсный список вуза.
- Deferred-acceptance симуляция (`backend/services/simulation.py`) +
  Monte Carlo слой поверх (`backend/services/monte_carlo.py`) — оценивает
  риск того, что часть абитуриентов без согласия его подадут, а часть с
  согласием передумает и уйдёт в другой вуз.
- Планировщик — 2 автоматических прогона в день
  (`backend/workers/scheduler.py`).
- После инцидента 2026-07-06 добавлен startup catch-up: если backend
  перезапущен после уже прошедшего слота `09:00`/`21:00`, а свежего
  успешного прогона за этот слот нет, планировщик ставит немедленный
  одноразовый запуск (`tests/test_scheduler.py`).

### Бауманка — полностью автоматизировано (задеплоено 2026-07-05)
- Публичный (без логина) JSON API Госуслуг «Вуз-навигатор»:
  `/api/university-applicant-list/v1/public/2026/competition/{groupId}/applicants`
  (`backend/services/bauman_scraper.py`).
- `groupId` нельзя получить одним запросом — собраны вручную через браузер
  (открыть `/vuznavigator/specialties/{oksoCode}/{levelId}/26`, перехватить
  `competition/statuses/ratings` в network, подобрать нужный `groupId` по
  applicationId Вари). Список — в `backend/config.py` (`BAUMAN_GROUP_IDS`).
- Подтверждены (по applicationId) 2 направления:
  - 27.03.05 Инноватика, `groupId=128989`, приоритет 1
  - 09.03.01 Информатика и вычислительная техника, `groupId=128864`, приоритет 2
- **Открытый вопрос:** пользователь ранее упоминал до 6 направлений в
  Бауманке (профили ИУ5/ИУ6/РК6 под 09.03.01 и отдельно 38.03.05
  Бизнес-информатика, ИБМ6). Прямой проверкой applicationId по всем
  `groupId` под известными `oksoCode` подтверждены только эти 2 — если она
  подавала документы ещё куда-то, нужно повторить ту же технику для
  оставшихся направлений.

### Фронтенд
- Одна длинная страница на все вузы (не вкладки) — у каждого вуза свой
  цвет и липкий баннер-разделитель («территория»), плюс полноширинная
  полоса прогресса сверху (градиент по вузам + маркер текущей позиции
  скролла).
- Сводка вверху страницы — кликабельный список, прыгает к разделу вуза.
- Для МАИ добавлен переключатель источника `МАИ` / `Госуслуги`.
  По умолчанию показывается официальный список МАИ; источник
  `МАИ Госуслуги` не выводится отдельной третьей секцией, а перерисовывает
  карточки, графики и историю внутри блока МАИ.
- Сайт теперь может работать как статический HTML со вшитым snapshot:
  `backend/services/snapshot.py` собирает данные тех же API-функций и
  записывает готовый `index.html` в `POSTUPI_WEB_ROOT`. Это было сделано,
  чтобы мобильный Safari/iPhone не зависал на живых API-запросах при
  открытии страницы.

## Известные особенности модели (не баги)
- Источники МАИ расходятся по смыслу данных. На 2026-07-06 priem.mai.ru
  отдавал срез `p20260704115255` с датой обновления `04.07.2026 14:36:52`,
  а Госуслуги уже показывали более широкий и свежий поток заявлений.
  Пример для ИВТ: МАИ `179 из 205`, Госуслуги `268 из 155`. Поэтому
  Госуслуги используются как отдельный переключаемый источник, а не как
  безусловная замена официального списка МАИ.
- «Вероятность зачисления» (Monte Carlo, `standalone_probability_pct`)
  может быть намного ниже «реального места», если у направления огромный
  пул подавших без согласия (`no_consent_count`) относительно мест —
  модель закладывает риск, что многие из них потом подтвердятся. Пример:
  Бауманка/ИВТ — реальное место 35 из 83 при 204 местах, но вероятность
  2%, так как ещё 1743 человека подали документы без согласия.
- Аналогично `предиктированный проходной балл` может выглядеть аномально
  (например, очень низким) для популярных направлений с большим пулом
  неподтверждённых заявок — короткие/обрывающиеся линии на графике
  «Прогноз проходного балла» означают «мало кто подал согласие, оценка
  ненадёжна», это отражено подсказкой под графиком.

## План Переделки На Полный Каскад

Обновлено: 2026-07-10, MSK.

### Почему Нужна Переделка

Текущий `Postupi` считает симуляцию только по тем направлениям, которые были
добавлены в его собственный прогон. Для источника `МАИ Госуслуги` это сейчас
ограниченный набор из `MAI_GOSUSLUGI_TARGET_CODES`:

- `09.03.01`;
- `09.03.02`;
- `09.03.03`;
- `27.03.03`;
- `10.05.02`.

Из-за этого модель видит не весь каскад МАИ. Конкуренты, которые в полном
каскаде уходят на другие направления МАИ, в старой модели могут оставаться
внутри Вариного набора и искусственно ухудшать её место.

Проверенный пример: `1422086`, `МАИ Госуслуги`, `09.03.03 Прикладная
информатика`, данные от 2026-07-10.

- В сыром списке: место `290`, балл `277`, приоритет `3`, согласие есть.
- Выше в сыром списке: `289` заявлений.
- Из них без согласия: `254`.
- С согласием: `35`.
- В полном каскаде из этих `35` согласившихся выше `30` уходят на другие
  направления, `5` остаются в Прикладной информатике.
- Старый ограниченный каскад `Postupi` даёт `20 из 19`.
- Полный каскад по данным нового проекта `C:\Projects\mai` даёт проходное
  положение: выше по баллу остаются только два человека, а при сценарии
  «Прикладная информатика как первый/единственный активный вариант» Варя
  получается `4 из 15` реально ведущихся туда при `19` местах.

Вывод: ошибка находится не в deferred acceptance и не во фронтенде, а во входном
universe направлений для симуляции.

### Целевая Архитектура

Новый проект `C:\Projects\mai` должен стать источником актуальных полных данных:

- `mai_snapshots`;
- `mai_competition_groups`;
- `mai_applicant_rows`;
- `bauman_snapshots`;
- `bauman_competition_groups`;
- `bauman_applicant_rows`.

`Postupi` должен использовать эти полные срезы для текущей симуляции, а свою
старую базу — как исторический слой для графиков. Старые расчёты нельзя
смешивать с новыми без пометки версии модели.

### План Работ

1. Добавить в `Direction` внешний идентификатор конкурсной группы:
   `external_group_id`.

2. Изменить уникальность направлений с `(university, name)` на связку вида
   `(university, external_group_id)`. Это важно, потому что разные конкурсные
   группы Госуслуг могут иметь одинаковое `oksoName`.

3. Добавить импорт полного среза из новой БД `mai` в `Postupi`, например
   сервис `backend/services/full_snapshot_import.py`.

4. Для МАИ импорт должен брать latest `ok` из `mai_snapshots`, все группы из
   `mai_competition_groups` и все строки из `mai_applicant_rows`.

5. Для Бауманки импорт должен брать latest `ok` из `bauman_snapshots`, все
   группы из `bauman_competition_groups` и все строки из
   `bauman_applicant_rows`.

6. Импорт должен создавать новый `MonitorRun`, обновлять/создавать
   `Direction`, писать `SeatPlan`, `CompetitorSnapshot`, логировать изменения
   у отслеживаемого абитуриента и запускать существующий `compute_simulation`.

7. `MAI_GOSUSLUGI_TARGET_CODES` и `BAUMAN_GROUP_IDS` больше нельзя использовать
   как фильтр входных данных для симуляции. Их можно оставить только как
   UI-фильтр или как legacy-настройку старого импортера.

8. Разделить «что считаем» и «что показываем»: симуляция должна видеть все
   конкурсные группы вуза, а карточки и графики должны показывать только те
   направления, где есть отслеживаемый абитуриент.

9. Сохранить старую историю `Postupi` как legacy-слой:
   `model_version=legacy_partial`, `coverage=tracked_scope_partial`.

10. Для новых полных расчётов использовать отдельные метки:
    `model_version=full_cascade_v1`, `coverage=full`.

11. На графиках разделить старые и новые серии:
    старая модель — пунктиром/приглушённо, новая полная модель — основной
    серией. В tooltip показывать `model_version`.

12. Переносить из старой истории как надёжные поля: сырой `position`,
    `total_score`, `priority`, `consent`, время прогона.

13. Поля `real_competitor_position`, `probability_pct`,
    `predicted_cutoff_score` из старой базы переносить только как legacy, не
    сравнивая напрямую с новой полной моделью.

14. Добавить тесты на кейс, где конкурент выше Вари уходит на направление вне
    Вариного списка. Ожидаемое поведение: полный каскад не считает такого
    конкурента мешающим целевому направлению.

15. Перед деплоем сделать бэкап продовой БД `Postupi`, прогнать новый импорт
    на копии, сверить ключевые кейсы и только потом пересобрать статический
    `postupi.tarko.su`.

собственных приоритетов абитуриента. Риск-прогноз Monte Carlo остаётся отдельной
строкой и больше не должен перекрашивать такое направление в `не проходит`.

## Учёт Дедлайна Подачи Заявлений На Бюджет (2026-07-13)

### Что Изменено В Модели

Monte Carlo слой больше не считает риск новых сильных заявлений постоянным до
конца кампании. Для бюджетного приёма добавлен дедлайн:

```text
2026-07-25T17:00:00+03:00
```

В `backend/services/monte_carlo.py` риски разделены на две части:

- перераспределение уже известных абитуриентов по согласиям, приоритетам и
  каскаду;
- появление новых потенциальных заявлений до дедлайна.

Новые потенциальные конкуренты добавляются в симуляцию только до дедлайна.
Ожидаемое число новых заявлений затухает через коэффициент:

```text
time_left_ratio = max(0, deadline_at - now) / max(1, deadline_at - campaign_start_at)
new_applicant_risk_factor = time_left_ratio ^ 0.7
```

После `25.07.2026 17:00 МСК`:

```text
new_applicant_risk_factor = 0
expected_new_applicants = 0
```

В `simulation_results` добавлены поля:

- `application_deadline_at`;
- `time_left_ratio`;
- `new_applicant_risk_factor`;
- `expected_new_applicants`.

`ensure_schema()` добавляет эти поля в существующую БД. Для PostgreSQL
`application_deadline_at` создаётся как `TIMESTAMP WITH TIME ZONE`; первый
вариант с `DATETIME` был исправлен после проверки на VPS, потому что PostgreSQL
не знает такой тип.

### Что Изменено В API И Интерфейсе

- `GET /postupi/status` и `GET /postupi/history` отдают новые поля дедлайна и
  коэффициента риска.
- В карточке рядом с прогнозом проходного выводится пояснение:

```text
Прогноз учитывает завершение подачи заявлений на бюджет: 25.07.2026 17:00 МСК
```

### Важное Ограничение По Местам 30/19

Разбор файла `Списки поступающих Прикл мат.html` от 2026-07-13 показал, что
для `09.03.03 Прикладная информатика`:

- `30` — общий КЦП на бюджет;
- `19` — основные места общего конкурса после вычитания квот:
  `30 - 3 особая - 3 отдельная - 5 целевая`.

Сейчас модель корректно учитывает дедлайн подачи заявлений, но отдельная
доработка ещё нужна для официального источника МАИ: ёмкость общего конкурса
должна считаться не просто по общему КЦП `30`, а сценарно с учётом БВИ,
особой, отдельной и целевой квоты, а также возврата незаполненных квот после
приоритетного этапа.
